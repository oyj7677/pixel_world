# Friend Pixel Log 전체 기능 계획서

작성일: 2026-05-16  
대상 프로젝트: Pixel World  
계획 성격: 제품 방향 전환 + 전체 기능 로드맵 + 자체 검토  
권장 실행 방식: 모든 기능을 한 번에 출시하지 않고, 전체 비전은 유지하되 단계별 MVP로 검증

---

## 1. 한 줄 방향

Pixel World는 r/place류의 공개 픽셀 캔버스에서 출발해, 친구들이 하루의 흔적을 픽셀로 남기고 그 과정을 리플레이/영상처럼 공유하는 **친구 기반 Daily Pixel Log 서비스**로 확장한다.

---

## 2. 핵심 문제 정의

기존 공개 픽셀 캔버스는 이미 강한 경쟁자가 있다.

- r/place: 대규모 커뮤니티 이벤트와 인터넷 밈
- Pxls: 지속형 픽셀 캔버스
- Wplace: 세계지도 기반 대규모 픽셀 경쟁

Pixel World가 단순 공개 캔버스로만 가면 차별화가 약하다. 반면 친구 기반으로 전환하면 다음 차별점이 생긴다.

- 친구 초대를 통한 자연 유입
- 하루 단위 기록성
- 공동 완성의 감정적 가치
- 결과물을 공유하는 바이럴 루프
- 공개 캔버스 대비 낮은 moderation 부담
- “그림 실력”보다 “참여 흔적” 중심의 낮은 진입 장벽

---

## 3. 제품 비전

### 비전 문장

> 친구들과 오늘의 흔적을 한 칸씩 남기고, 하루가 끝나면 우리만의 픽셀 리플레이가 완성된다.

### 사용자에게 보이는 약속

- 친구 방에 들어간다.
- 오늘의 색 또는 감정을 선택한다.
- 한 칸을 남긴다.
- 친구들이 남긴 픽셀이 쌓인다.
- 하루가 끝나면 완성 과정이 리플레이로 재생된다.
- 결과물을 링크나 영상으로 공유한다.

---

## 4. 제품 원칙

1. **첫 참여는 3초 안에 끝나야 한다.**  
   사용자가 처음부터 픽셀 아트를 고민하게 만들면 안 된다.

2. **픽셀은 작업이 아니라 흔적이어야 한다.**  
   “그림을 그린다”보다 “친구의 오늘에 한 칸 남긴다”가 핵심이다.

3. **모든 고급 기능은 기본 경험 뒤에 숨긴다.**  
   직접 그리기, 방해, 팀전, 템플릿, 영상 export는 초기 화면을 복잡하게 만들지 않는다.

4. **친구 초대가 제품 루프의 중심이다.**  
   방 생성 → 초대 → 참여 → 리플레이 → 공유 → 새 방/새 친구 초대가 반복되어야 한다.

5. **운영/신고/초기화는 처음부터 준비한다.**  
   친구 기반이라도 부적절한 콘텐츠와 악용 가능성은 존재한다.

---

## 5. 핵심 사용자 루프

### Daily Friend Loop

1. 사용자가 오늘의 방을 만든다.
2. 초대 링크를 친구에게 보낸다.
3. 친구가 닉네임으로 입장한다.
4. “내 색으로 한 칸 남기기”를 누른다.
5. 친구들의 픽셀이 쌓인다.
6. 하루가 끝나면 리플레이가 생성된다.
7. 완성 결과를 공유한다.
8. 다음 날 다시 새 로그가 열린다.

### Engagement Loop

- 참여하지 않은 친구에게 가벼운 알림
- 모두가 한 번씩 참여하면 뱃지
- 친구별 색상/기여도 표시
- 어제 리플레이를 오늘의 초대 이유로 재활용

### Advanced Loop

- 직접 그리기 모드
- 팀전/방해 모드
- 복구 아이템
- 일일 미션
- 이벤트 방
- 템플릿 챌린지

## 6. 전체 기능 목록

### A. 계정/방/초대

#### A1. 닉네임 기반 빠른 입장

- 로그인 없이 초대 링크로 방 입장
- 최초 입장 시 닉네임 입력
- 쿠키 기반 임시 사용자 식별
- 나중에 계정 전환 가능

#### A2. 방 생성

- 방 이름 입력
- 방 유형 선택
  - Daily Log
  - Free Canvas
  - Pixel War
  - Event Room
- 캔버스 크기 선택
  - 16×16
  - 32×32
  - 50×50
  - 100×100

#### A3. 초대 링크

- 방별 공유 링크 생성
- 링크 복사 버튼
- 카카오톡/인스타/문자 공유용 문구
- 초대한 사람 표시

#### A4. 친구 목록

- 자주 함께하는 친구 목록
- 최근 참여한 방 기준 추천
- 향후 계정 시스템과 연동

---

### B. Daily Pixel Log

#### B1. 하루 단위 캔버스

- 방마다 오늘의 캔버스 생성
- 하루가 끝나면 마감
- 마감 후 수정 불가
- 다음 날 새 캔버스 자동 생성

#### B2. Quick Pixel

- “내 색으로 한 칸 남기기” 버튼
- 추천 위치 자동 선택
- 직접 위치 선택은 보조 옵션
- 첫 참여 시 색상 자동 추천

#### B3. 픽셀 스택

- 일정 시간마다 픽셀 권한이 쌓임
- 예: 3시간마다 1개, 최대 6개 보관
- 바쁜 사용자는 나중에 한 번에 참여 가능
- 셋로그식 강제 시간 압박을 줄이는 장치

#### B4. 오늘의 상태/감정 색

- 오늘 기분 선택
- 선택한 감정이 색상으로 변환
- 예: 행복, 피곤, 집중, 설렘, 우울, 화남

#### B5. 친구별 색상

- 친구마다 기본 색 지정
- 리플레이에서 누가 찍었는지 표시
- 방장이 색상 규칙을 선택 가능

---

### C. 캔버스 편집 경험

#### C1. 기본 픽셀 찍기

- 현재 구현된 실시간 픽셀 배치 유지
- 방/일자 단위 canvasId로 확장
- 쿨타임 또는 픽셀 스택 정책 적용

#### C2. 직접 그리기 모드

- 고급 사용자용 모드
- 위치와 색상을 직접 선택
- 친구들과 맞춰 그림 완성 가능

#### C3. 자동 배치 모드

- 빈 공간 우선
- 친구들이 많이 찍은 주변 추천
- 템플릿이 있으면 템플릿의 미완성 구역 추천

#### C4. 되돌리기/복구

- 방장 또는 운영자만 특정 영역 초기화 가능
- 최근 N분 내 본인 픽셀 취소 가능 여부 검토
- 악용 방지를 위해 제한 적용

---

### D. 리플레이/영상화

#### D1. 웹 리플레이

- 픽셀이 찍힌 순서대로 재생
- 속도 조절
- 친구별 색상/닉네임 표시
- 특정 친구만 하이라이트

#### D2. 공유 페이지

- 방/일자별 결과물 공개 링크
- 공개/비공개 설정
- 완성 이미지와 리플레이 보기

#### D3. 영상 export

- 초기에는 웹 리플레이만 제공
- 이후 서버 또는 브라우저에서 mp4/gif 생성
- 인스타/틱톡용 9:16 세로 템플릿
- 마지막 프레임에 참여자 닉네임 표시

#### D4. 하이라이트 자동 생성

- 가장 많이 찍은 사람
- 마지막 픽셀을 찍은 사람
- 가장 많이 바뀐 영역
- 완성 직전/직후 비교

---

### E. 소셜/바이럴 기능

#### E1. 참여 상태

- 오늘 참여한 친구
- 아직 참여하지 않은 친구
- 모두 참여 시 완료 배지

#### E2. 가벼운 알림

- 방장이 친구에게 참여 요청
- 강한 푸시보다 링크/공유 중심
- 알림 피로도를 줄이기 위해 빈도 제한

#### E3. 공유 문구

- “오늘 우리 방에 한 칸 남겨줘”
- “네 색 없으면 오늘 로그가 비어 있어”
- “우리 하루가 픽셀로 완성됐어”

#### E4. 기여도 카드

- 친구별 픽셀 수
- 친구별 대표 색
- 오늘의 마지막 픽셀
- 이번 주 연속 참여

---

### F. 게임화/체류 기능

#### F1. 일일 미션

- 오늘 1픽셀 남기기
- 친구 3명 참여시키기
- 특정 색상으로 5칸 채우기
- 리플레이 공유하기

#### F2. 배지

- 첫 방 생성
- 첫 리플레이 완성
- 7일 연속 참여
- 모두 참여한 방
- 가장 빠른 참여자

#### F3. 방해/복구 모드

- 일반 Daily Log에는 기본 비활성화
- Pixel War 모드에서만 활성화
- 방해 픽셀, 보호 픽셀, 복구 아이템 등

#### F4. 팀전

- 친구 그룹을 팀으로 나눔
- 제한 시간 동안 영역 점령
- 리플레이로 승패 과정 확인

---

### G. 템플릿/이벤트

#### G1. 템플릿 방

- 생일
- 여행
- 시험 끝
- 프로젝트 완료
- 커플/친구 기념일
- 회사/팀 회고

#### G2. 시즌 이벤트

- 크리스마스 픽셀 트리
- 새해 목표 캔버스
- 방학/졸업/입학 이벤트

#### G3. 이미지 기반 템플릿

- 이전에 검토했던 Pixelizer는 메인 기능에서 제거했지만, 추후 템플릿 생성 도구로 제한적으로 재도입 가능
- 사용자가 이미지를 업로드하면 방 템플릿으로 변환
- 일반 사용자용 핵심 루프와 분리

## 8. 단계별 실행 계획

## Phase 0: 현재 MVP 정리

### 목표

현재 전역 캔버스 MVP를 안정화하고, 방/일자 확장을 위한 기반을 정리한다.

### 포함 기능

- 현재 실시간 캔버스 유지
- 관리자 영역 초기화 유지
- 개인 Recent 유지
- 부하 테스트 스크립트 유지
- Pixelizer 제거 상태 유지

### 완료 기준

- `npm run verify` 통과
- `/` 페이지 정상
- `/admin` 페이지 정상
- 관리자 영역 초기화 후 실시간 반영 확인

---

## Phase 1: 친구 방 MVP

### 목표

공개 전역 캔버스에서 친구 방 기반 제품으로 전환하는 첫 단계.

### 포함 기능

- 방 생성
- 초대 링크
- 닉네임 입장
- 방 단위 캔버스
- 방별 실시간 픽셀 동기화
- 방별 관리자/방장 최소 권한

### 제외 기능

- 영상 export
- 팀전
- 방해 아이템
- 프리미엄 결제

### 성공 기준

- 사용자가 30초 안에 방을 만들고 링크를 공유할 수 있다.
- 초대받은 사용자가 10초 안에 닉네임 입력 후 첫 픽셀을 남길 수 있다.
- 한 방에 10명이 접속해도 픽셀 업데이트가 실시간 반영된다.

---

## Phase 2: Daily Pixel Log

### 목표

하루 단위 기록 제품으로 전환한다.

### 포함 기능

- daily_canvases 도입
- 하루 단위 마감
- Quick Pixel
- 픽셀 스택
- 오늘의 감정 색
- 참여자 목록

### 성공 기준

- 초대받은 사용자의 첫 픽셀 전환율 60% 이상을 목표로 한다.
- 첫 참여 과정은 닉네임 입력 후 3초 이하의 의사결정으로 완료 가능해야 한다.

## 10. 기술 설계 방향

### 현재 자산 재사용

- Next.js 웹 앱
- Fastify API 서버
- Socket.IO 실시간 동기화
- PostgreSQL pixel/event 저장
- Redis cooldown/store
- 관리자 로그인/영역 초기화
- 부하 테스트 스크립트

### 주요 확장

- canvasId를 global에서 room/daily canvas 단위로 확장
- Socket.IO room join 구조 도입
- pixel_events를 replay 원본 데이터로 활용
- Recent는 개인/방/리플레이 목적에 따라 조회 범위 분리
- 관리자 API는 roomId/canvasId scope를 필수로 받도록 확장

### 성능 고려

- 작은 친구 방은 실시간 이벤트 그대로 처리
- 대형 이벤트 방은 Redis adapter 또는 pub/sub 확장 고려
- 리플레이는 이벤트를 매번 full scan하지 않고 snapshot/cache 도입 검토
- 영상 export는 초기에는 비동기 작업 큐 필요

## 11. 위험과 대응

### 위험 1: 기능이 너무 많아 첫 경험이 복잡해진다

- 공유 문구 A/B 테스트
- 리플레이를 초대 이유로 재활용
- “네 색이 없으면 비어 있어” 같은 사회적 동기 부여

대응:

### 위험 5: 영상 export 비용과 구현 복잡도가 크다

대응:

- 초기에는 웹 리플레이만 제공

## 14. 자체 검토 결과

### 검토 Verdict

**조건부 승인.**  
전체 방향은 차별화 가능성이 높지만, “모든 기능”을 한 번에 출시하려고 하면 제품의 첫 경험이 무너질 위험이 크다. 따라서 전체 기능은 장기 로드맵으로 보관하고, 실행은 Phase 1~3까지만 먼저 집중하는 것이 적절하다.

### 강점

- r/place/Wplace와 다른 친구 기반 포지션이 명확하다.
- 현재 구현된 실시간 캔버스, 관리자, 이벤트 로그를 재사용할 수 있다.
- 리플레이/공유 구조가 바이럴 루프를 만들 수 있다.
- Quick Pixel을 통해 첫 진입 부담을 낮출 수 있다.

### 약점

- “픽셀을 찍는 행위” 자체가 영상 촬영보다 인지 부담이 크다.
- 친구 초대가 실패하면 제품 루프가 약해진다.
- 영상 export는 구현 비용과 서버 비용이 크다.
- 방/일자/멤버/권한이 들어오면 데이터 모델이 복잡해진다.

### 보완 필요

- Phase 1 이전에 첫 참여 화면 와이어프레임이 필요하다.
- Quick Pixel의 자동 위치 추천 규칙을 구체화해야 한다.
- 방 공개/비공개 기본값을 정해야 한다.
- 리플레이를 웹 기반으로 먼저 할지, 이미지 시퀀스로 저장할지 결정해야 한다.
- 계정 없이 시작하되 나중에 계정으로 연결하는 migration 전략이 필요하다.

### 가장 큰 제품 리스크

> 사용자가 “재밌다”보다 “귀찮다”를 먼저 느끼는 순간 이 제품은 실패한다.

따라서 첫 버전은 기능 수보다 진입 부담 최소화가 더 중요하다.

## 16. 발표용 요약

Pixel World는 처음에는 공개 픽셀 캔버스로 시작했지만, 경쟁 서비스와 차별화하기 위해 친구 기반 Daily Pixel Log 서비스로 방향을 확장한다.

핵심은 사용자가 그림을 잘 그리는 것이 아니라, 친구의 하루에 자신의 색 한 칸을 남기는 것이다. 하루가 끝나면 친구들이 남긴 픽셀이 리플레이로 재생되고, 이 결과물이 다시 공유와 초대의 이유가 된다.

이 방향은 기존 실시간 캔버스 기술을 재사용하면서도, 공개 대규모 캔버스와 다른 감정적/소셜 가치를 만들 수 있다.
